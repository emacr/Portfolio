    context: "The system needed decoupled communication between Order Service and Notification Service. RabbitMQ and Kafka were evaluated for the projected message volume."
    decision: "RabbitMQ with durable queues and dead-letter queues was chosen. Projected volume (< 5K msg/s) did not justify Kafka's operational complexity and team learning curve."
    consequences: "Lower operational and deployment complexity. Trade-off: if volume grows 10x, migration to Kafka will be required. Migration trigger metrics were documented."
  - title: "ADR-002: Per-service PostgreSQL vs. shared database"
    context: "A shared database (simpler) vs. per-service database (Database-per-Service pattern) was debated to ensure real decoupling between bounded contexts."
    decision: "Each microservice has its own PostgreSQL instance in a dedicated Kubernetes namespace. Cross-service communication happens exclusively via API/events."
    consequences: "Strong fault isolation and independent schema scaling. Cost: distributed transactions require the Saga pattern, increasing rollback complexity."
  - title: "ADR-003: Kong API Gateway over custom implementation"
    context: "The API Gateway needed to handle rate limiting, OAuth2 authentication, and centralized logging. Building it in Express.js vs. using Kong Gateway was evaluated."

    consequences: "Faster delivery and proven production robustness. Vendor dependency managed with versioned helm charts for reproducibility."

## Project Context

This project solved the scalability problem of a monolithic platform hitting its limits during high-demand events (Black Friday, product launches). The migration to microservices allowed scaling critical services independently.

## Key Technical Challenges

The main challenge was the **incremental migration** from the monolith, using the Strangler Fig pattern to extract services one by one without interrupting production operations over 6 months.
